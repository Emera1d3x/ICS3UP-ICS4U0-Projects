Object Oriented Programming (shortened as OOP) is a programming concept based on real-world objects. The belief is that we can effectively create anything we see in the real world in code and better. Some of its main concepts (which were focused on in this course) are Encapsulation, Inheritance, Polymorphism, and Abstraction. These ideas are a foundation, and object oriented programming languages such as Java‚òï, Pythonüêç, C++ ¬©Ô∏è or C# #Ô∏è‚É£support this with memory allocation, references and effectively a complex system of transisters that take in input, and produce output.

Why might one even go through the hassle of creating so called "‚ñ≥ objects ‚óé"? Benefits include code-reusability, code-maintainability, a well-structured system, security, flexibility, and more.

First, let us take a very brief look at what the four main OOP topics taught in this course means conceptually.

- Encapsulation

    Encapsulation refers to having variables (attributes) and methods (functions/abilities) bundled up into a single unit (known as a class in Java) which may have the ability to be private and/or gatekeep certain information. It helps achieve data-hiding. Its internal details are hidden from outside itself, but it allows for interactions which are black-boxed and kept safe.

- Inheritance

    Inheritance is the ability for a child object to "inherit" or get passed on traits, attributes, and abilities from a "parent" object. This enables the newly derived class to have the relationship "is-a" with its base class and add-on additional information.

- Polymorphism

    Polymorphism is the ability for something to have different forms. In computer science, it is specifically children classes implementing and morphing their own unique features with the parent class. In Java, it is said that there is an Object class at the top of the class hierarchy. From that, there are children all of which have unique features some concrete, some abstract, which together creates the programs that we use every day.

- Abstraction

    Abstraction is the idea that you can have literally anything as an object. You can even have abstract and intangible ideas, these may be abstract classes or interfaces. Abstract classes and interfaces have variables and methods, but they are not literally defined. (A method may exist, but its body/content are empty). This gives programmers the ability to start with any concept they can think of and work their way "down" by slowly adding definitions to their idea in subclasses. This promotes code-reusability, code-maintainability, organization and helps code become scalable.

    Encapsulation Example

As stated previously, Encapsulation is essentially the ability for a chunk of code (in this case a class) to be self-contained and have data hiding. The class allows for some information to be directly given while others to be protected. The main use is to ensure the core contents of the class is never obstructed by external sources for safety. Lets adjust our Bird class to implement this idea!

Let us begin by ensuring all variables are private!

Now all variables are not directly accessible from external sources. Private class variables may be accessed from everywhere within its own class but not from others.

    Polymorphism Example

As we know from inheritance, SubClasses inherit the functions and attributes of their SuperClass. The idea of Polymorphism and its implementations in Java build on this and allow SubClasses to redefine what a SuperClass method does. This essentially allows a singular object to have multiple forms depending on the level of what the object it is.

Say we have a SuperClass Mammal and SubClass Bird. (Refer to the organism tree I made at the bottom of Inheritance). All (technically most) mammals can travel in some shape or form. For simplicity purposes say a mammal can travel by foot. The subclass Bird may redefine travelling with its special wings, but it can also travel by foot.

When a Bird object is asked to travel, it will default to travelling by flight, but if the Bird class allows itself to travel using its superclass, it can via the keyword super, a reference to the object's parent.

Lets see this in code. Here is a Mammal class:


        File Name: Mammal.java

    1     public class Mammal extends Animal {
    2       // Attributes / Variables
    3       // Functions / Methods
    4       public void travel(){
    5           System.out.println("Travelling by foot");
    6       }
    7     }

Here is the Bird subclass showcasing how it may redefine the travel method.


        File Name: Bird.java

    1     public class Bird extends Mammal {
    2       // Attributes / Variables
    2       // Functions / Methods
    3       public void travel(){
    4           System.out.println("Travelling with wings");
    5       }
    6     }

Now when a bird object performs the travel functions, this happens:


        (IN ANOTHER FILE)

    1     Bird var = new Bird();
    2     var.travel()

        (Output Console)
    1     Travelling with wings

The technique of a SubClass redefining a SuperClass is called OVERRIDING.

Here is a Bird class that instead calls its SuperClass's method, then adds its unique function. It calls the SuperClass method via the keyword "super", thus running the SuperClass's method first:


        File Name: Bird.java

    1     public class Bird extends Mammal {
    2       // Attributes / Variables
    2       // Functions / Methods
    3       public void travel(){
    4           super.travel();
    5           System.out.println("Travelling with wings");
    6       }
    7     }

Here is a demonstration:


        (IN ANOTHER FILE)

    1     Bird var = new Bird();
    2     var.travel()

        (Output Console)
    1     Travelling by foot
    2     Travelling with wings

The "super" keyword essentially calls the SuperClass of the class it is in.

Some additional things you need to know are:

  • If you have
    SuperClass var = new SubClass();
    var may NOT use SubClass methods unless it is downcasted or is a overridden method. The overridden method only works because the method name exists within SuperClass allowing it to compile during compile time, then at runtime recognizes the SubClass overrider.
  • A SubClass may not create a SuperClass.
  • If a SuperClass calls a method within itself, the SuperClass's method is used, not the SubClass's, even if the name is overridden.

    Abstraction Example

As stated previously, Abstraction is the idea that classes may be abstract. Classes may have predetermined method names, but no definition/code within the method - making it essentially useless. To add use to such a vaguely defined class, it has to be extended with SubClasses which purposefully override the methods and gives it functions.

There are many different variable names and method names, but no definition or true function. The "abstract" keyword ensures that this class may not be instantiated and allows for different syntax.
